You are a senior AI systems architect and backend engineer.

Create/update the `ai-core` branch and implement the AI Core microservice for a real-time interview/practice platform.

Branch:
`ai-core`

Primary goal:
Build one reusable AI Core service that can power both:

1. Full mock interviews
2. Smaller custom practice sessions

Do NOT build separate services for each mode.

---

## HIGH-LEVEL PRODUCT CONTEXT

This system has another microservice that already prepares a research/context file from user inputs.

That upstream service takes:

* resume
* job description
* target company
* role type
* user preferences
* internet research
* Reddit/interview threads
* previous interview patterns
* company/role expectations

It outputs a structured context file.

This AI Core service receives that context file and uses it to run adaptive interview/practice sessions.

---

## CORE DESIGN PRINCIPLE

Use one central orchestrator.

Do NOT create a messy system where many agents talk independently.

Use:

* one InterviewOrchestrator
* one SessionPlanner
* helper modules/sub-agents for specific tasks

---

## SUPPORTED SESSION TYPES

The same service must support:

1. FULL_INTERVIEW

* Default 60 minutes
* 25 min behavioral + resume + technical discussion
* 35 min coding round

2. BEHAVIORAL_PRACTICE

* Short targeted practice
* Example: “Tell me about yourself”
* Example: STAR story practice
* Example: conflict/teamwork/leadership questions

3. TECHNICAL_CONCEPT_PRACTICE

* Practice explaining one concept
* Example: Redis caching
* Example: database indexing
* Example: microservices
* Example: distributed systems

4. CODING_PRACTICE

* LeetCode-style coding round
* Candidate explains approach, codes, handles follow-ups

5. RESUME_DEEP_DIVE

* Interviewer probes one or more resume projects
* Focus on ownership, tradeoffs, implementation details

6. CUSTOM_QUESTION

* User provides a custom question or focus area
* System conducts a focused mini-interview around it

---

## SESSION INPUT

The service should accept a session configuration like:

{
"session_type": "BEHAVIORAL_PRACTICE",
"duration_minutes": 15,
"mode": "learning",
"focus_area": "tell me about yourself",
"context_file": "...",
"resume": "...",
"job_description": "...",
"company": "Akamai",
"role_type": "SDE1",
"difficulty": "medium"
}

`context_file` is the research document generated by the upstream microservice.

It may contain:

* resume summary
* JD summary
* company summary
* role expectations
* common interview questions
* likely technical topics
* Reddit/interview thread insights
* behavioral themes
* coding patterns likely for this company/role

## MAIN COMPONENTS

1. Context Loader

* Accept context file input
* Parse and normalize it
* Extract:

  * candidate profile
  * target role
  * company expectations
  * likely interview topics
  * resume/project highlights
  * risk areas to probe
  * suggested questions

2. SessionPlanner

* Converts session_type + duration + focus_area + context into a session plan
* Chooses phases
* Allocates time
* Chooses question strategy

Examples:

FULL_INTERVIEW:

* INTRODUCTION
* RESUME_DEEP_DIVE
* BEHAVIORAL
* TECHNICAL_DISCUSSION
* CODING_ROUND
* CODING_FOLLOWUPS
* FINAL_WRAP
* REPORT_GENERATION

BEHAVIORAL_PRACTICE:

* INTRODUCTION_OR_PROMPT
* BEHAVIORAL_RESPONSE
* FOLLOWUPS
* FEEDBACK

TECHNICAL_CONCEPT_PRACTICE:

* CONCEPT_EXPLANATION
* DEPTH_FOLLOWUPS
* TRADEOFFS
* FEEDBACK

CODING_PRACTICE:

* PROBLEM_STATEMENT
* APPROACH_DISCUSSION
* CODING
* EDGE_CASES
* COMPLEXITY
* FEEDBACK

3. InterviewOrchestrator

* Owns current session state
* Tracks current phase
* Tracks time remaining
* Maintains rolling conversation summary
* Decides what to ask next
* Calls helper modules
* Handles phase transitions
* Sends text to TTS pipeline

4. StreamingTranscriptHandler

* Receives transcript chunks while user speaks
* Updates live transcript buffer
* Maintains partial answer state
* Detects speech end events
* Sends partial context to QuestionGenerationModule while user is speaking

5. QuestionGenerationModule

* Runs in background while candidate is speaking
* Generates 3–5 possible follow-up questions using:

  * current phase
  * session type
  * focus area
  * latest partial transcript
  * conversation summary
  * context file
  * resume/JD/company context
* Updates candidate question list continuously

6. FollowupSelector

* Runs when speech_ended event occurs
* Makes a small, fast LLM call
* Selects best next question from candidates
* Optimized for low latency

7. ResponseGenerator

* Uses OpenAI reasoning model
* Generates final interviewer message
* Must stream response text as soon as possible
* Should be short, natural, and interviewer-like

8. VoiceService

* Use ElevenLabs TTS for interviewer voice
* Prefer streaming TTS
* Keep STT as a pluggable interface
* If ElevenLabs STT is available, use it
* Otherwise create abstraction for OpenAI/Whisper/other STT

9. Evaluator

* Scores candidate based on session type
* Generates feedback report
* For mini practice sessions, generate focused feedback
* For full interviews, generate comprehensive interview report

## API REQUIREMENTS

FastAPI endpoints:

POST /sessions/start

* accepts session config and context file
* creates session
* returns session_id and initial interviewer message

WebSocket /sessions/{session_id}/stream

* handles real-time events

POST /sessions/{session_id}/end

* ends session and triggers report generation

GET /sessions/{session_id}/report

* returns final report

POST /sessions/{session_id}/text-test

* mock endpoint for testing without audio
* accepts plain transcript text and returns next interviewer response
